Chart a route through a foggy problem: turn a loose goal into a MECE map of sub-questions,
then resolve them one at a time until the path to the answer is clear. This is consulting
**problem-structuring** (issue tree / hypothesis tree) — use it when there isn't a plan
*yet*. (To stress-test a plan you already have, use `/grill-me`.)

## 1. Frame the real question
Pin the actual question behind the fuzz, in one line. Reframe if needed ("you said X; the
decision is really Y"). State what "solved" looks like — the deliverable or decision it feeds.

## 2. Decompose (MECE)
Break the problem into **3-6 mutually-exclusive, collectively-exhaustive** sub-questions — an
issue tree. Prefer **hypothesis-led** branches where you can (a candidate answer per branch,
not just a topic). Keep it shallow first; deepen only the branch that matters.

## 3. Prioritise the route
Rank branches by **impact × ease**: which one most reduces uncertainty — or, if false,
collapses the whole thing? Name the "one thing that, if resolved, clears most of the fog." Start there.

## 4. Turn into a workplan
For the top branches, lay out: the sub-question · how you'd answer it (data / analysis / who
to ask) · rough effort · owner. Output an **ordered list of investigation steps**.

## 5. Resolve one at a time
Work the first step (delegate to `analyst` / `briefing-analyst` as fits), report the finding,
**update the map** (prune settled branches, re-rank the rest), then move on. Stop when the
path to the answer is clear — you don't have to resolve every branch, just enough to see the way.

File the map + turn the steps into tasks (backlog / Notion). Log durable structural calls to
`/decision-log`.
